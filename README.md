# FastAPI MongoDB GridFS Service
Generated by AI, except for this section, and a few bug fixes.

Use of cross-network non-directly connected database scenarios.
A lightweight, asynchronous API service built with FastAPI and Motor (async MongoDB driver) for standard MongoDB document operations and GridFS file management.
This project supports core MongoDB data CRUD, GridFS file upload/search/download/delete, and production deployment on Linux via systemd. No local MongoDB service dependency.

## Features

- Async MongoDB document operations (create/update/delete/query)

- Full GridFS file management: upload, search, download, delete

- FastAPI automatic interactive API docs (Swagger UI)

- Production-ready deployment with Gunicorn

- Linux systemd service for background running and auto-restart

- No dependency on local MongoDB service (supports remote MongoDB)

Create a `.env` file in the project root directory:

```env

API_V1_STR=/api/v1
DEBUG=False
```

## API Endpoints

All endpoints are prefixed with `/api/v1`. This is configurable in the .env file.
Accessing the "/docs" path allows you to view all the interface addresses.

### 1. Health Check

```http
GET /health
Response: {"status": "healthy", "message": "Service is running"}
```

### 2. Standard MongoDB Data Operations

#### Write Data (Create/Update/Delete)

```http
POST /api/v1/data/write
Content-Type: application/json
```

Supported operations:`create`, `update`, `delete`

Example Request Body:

```json
{
  "collection": "test_collection",
  "operation": "create",
  "filter": {"name": "test_item"},
  "data": {
    "name": "test_item",
    "value": 100,
    "description": "Sample data"
  }
}
```

#### Query Data

```http
POST /api/v1/query
Content-Type: application/json
```

Example Request Body:

```json
{
  "collection": "test_collection",
  "filter": {"value": {"$gt": 50}}
}
```

### 3. GridFS File Operations

#### Upload File

```http
POST /api/v1/gridfs/bucket/{bucket_name}/upload
Content-Type: multipart/form-data
```

#### Search Files

```http
POST /api/v1/gridfs/bucket/{bucket_name}/search
Content-Type: application/json
```

Example Request Body (empty {} for all files):

```json
{"filename": {"$regex": "test", "$options": "i"}}
```

#### Download File

```http
GET /api/v1/gridfs/bucket/{bucket_name}/download/{file_id}
```

#### Delete File

```http
DELETE /api/v1/gridfs/bucket/{bucket_name}/delete/{file_id}
```

## Linux Production Deployment

### 1. Gunicorn Configuration

Use `gunicorn_config.py` in project root:
The contents of this file allow for modification of IP and port bindings, as well as the number of working processes, etc.

### 2. Systemd Service File

Replace service file: `/etc/systemd/system/mongo-api.service`

Replace `/path/to/your/project` with your actual project absolute path

### 3. Enable and Start Service

```bash
# Reload systemd configuration
sudo systemctl daemon-reload

# Enable auto-start on boot
sudo systemctl enable mongo-api.service

# Start service
sudo systemctl start mongo-api.service

# Check service status
sudo systemctl status mongo-api.service

# View real-time logs
journalctl -u mongo-api -f
```

### 4. Common Service Commands

```bash
sudo systemctl stop mongo-api.service
sudo systemctl restart mongo-api.service
sudo systemctl disable mongo-api.service
```

## Troubleshooting

- **Async/GridOut errors**: Fixed in the latest gridfs_service.py (await stream and read bytes properly)
- **MongoDB connection failed**: Check MONGODB_URI in .env, ensure remote MongoDB allows external access
- **Service failed to start**: Verify project path and virtual environment path in systemd service file
- **Permission errors**: Ensure www-data user has read access to project directory
